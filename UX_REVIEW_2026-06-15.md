## Section 2 — TL;DR

**Green:** Architecture is solid. SwiftData models are correct (defaults, no @Relationship, UUID linking). Concurrency in networking/audio/weather services is clean (`await MainActor.run`, `Task { @MainActor in }`). Design token system (colors, fonts, spacing, animations) is well-established and mostly followed. TimelineView + delta-time physics correctly used in all game loops. `.trailingFade()` and `.texturedButton(tint:)` are consistently used in the core purchase/recipe flows.

**Red:** Three categories of violations appear throughout the codebase:

1. **P0 — Architecture Rule §2 (Concurrency):** `DispatchQueue.main.async` appears in 27+ sites across multiplayer managers, auth callbacks, and Game Center delegates — all should be `Task { @MainActor in }`. `try? save()` silently swallows SwiftData errors in 16 sites (March 2026 bug that destroyed child profiles was this exact pattern).

2. **P0 — Kid-lens tap targets:** PlotView bug/weed/water taps are 10–12pt emoji-on-circle — a child cannot reliably tap these. This is the most impactful UX issue in the whole game (it's the *core* garden interaction).

3. **P1 — Component reuse debt:** `PrimaryButtonStyle()` (legacy dead code) used in 5 onboarding views instead of `.texturedButton(tint:)`. Inline gender-image bypasses of `profilePoseImage` in 15+ call sites across multiplayer/avatar/onboarding views. 5+ horizontal carousels missing the mandatory `.trailingFade()` affordance.

**Count summary:**

## Section 3 — P0 Findings (Critical — ship blockers)

### P0-A: `DispatchQueue.main.async` — 27+ sites (Architecture Rule §2)

Every one of these must become `Task { @MainActor in }`. The `DispatchQueue` path bypasses Swift 6 strict concurrency and will produce data-race warnings (or crashes) under strict isolation. Prior art: `GardenWeatherService`, `PipGameAnimationView`, `CharacterWalkingView` all already use the correct pattern.

| File | Sites | Context |
|---|---|---|
| `MultiplayerManager.swift` | lines 65, 196, 240, 297, 321 | Game Center match callbacks |
| `NearbyMultiplayerManager.swift` | lines 155, 198, 221, 241, 286, 308 | MCSession delegate callbacks |
| `SessionManager.swift` | ~8 sites | Profile load, PIN verify, route transitions |
| `GameCenterService.swift` | line 102 | `authenticateHandler` completion |
| `GameCenterMatchmakerView.swift` | lines 42, 51, 59 | Coordinator delegate (cancelled, failed, found) |
| `AuthManager.swift` | line 113 | `checkExistingCredential()` ASCredentialState |
| `SeedInfoView.swift` | `updateUIView()` | UIViewRepresentable delegate |
| `ParentPINEntryView.swift` | line 134 | PIN verification callback |
| `ChefAcademyApp.swift` | 1 site | App-level route transition |

**Fix pattern (identical for all):**
```swift
// BEFORE
DispatchQueue.main.async {
    self.someState = newValue
}

// AFTER
Task { @MainActor in
    self.someState = newValue
}
```

For delegate callbacks where `self` is not `@MainActor`, annotate the surrounding class/method or use `await MainActor.run { }`.

### P0-B: `try? save()` — 16 sites (Architecture Rule §1)

Silent SwiftData save failures destroyed a week of child profile data in March 2026. This is documented in §1 as a **never-repeat** bug.

| File | Count |
|---|---|
| `SessionManager.swift` | 8 |
| `ParentDashboardView.swift` | 2 |
| `ChefAcademyApp.swift` | 1 |
| `FamilySetupView.swift` | 1 |
| `AddChildFlowView.swift` | 1 |
| `AllergenEditorSheet.swift` | 1 |
| `SiblingGardenView.swift` | 1 |
| `SiblingProfileView.swift` | 1 |
| **Total** | **16** |

**Fix pattern (identical for all):**
```swift
// BEFORE
try? modelContext.save()

// AFTER
do {
    try modelContext.save()
} catch {
    print("[Save error] \(error)")
}
```

### P0-C: PlotView — Tap targets 10–12pt (Anti-pattern D)

The bug, weed, and water interaction circles on `PlotView` are rendered as emoji on circles approximately 10–12pt in diameter with `.onTapGesture` — not wrapped in `Button`. This is the *core* garden interaction loop (PlotView is what kids spend the most time tapping). No child can reliably hit a 10–12pt target; Apple's HIG minimum and `AppSpacing.minTapTarget` both require 44pt.

**Current pattern:**
```swift
// PlotView — approximate reconstruction
Text(bugEmoji)
    .font(.system(size: 12))
    .onTapGesture { rescueBug() }
```

**Fix:**
```swift
Button(action: rescueBug) {
    Text(bugEmoji)
        .font(.AppTheme.title3) // or .title for visibility
}
.buttonStyle(BouncyButtonStyle())
.frame(minWidth: AppSpacing.minTapTarget, minHeight: AppSpacing.minTapTarget)
```

---

### P0-D: GardenView — DraggablePip hint text at 9pt

The "Drag me!" affordance label under the draggable Pip is rendered at 9pt system font — below the readable floor for a 6-year-old and below `Font.AppTheme.caption` (12pt). A child cannot read this, defeating the affordance entirely.

**Fix:** Use `Font.AppTheme.caption` at minimum; `Font.AppTheme.subheadline` preferred for a primary drag hint. Also consider replacing text with a gesture icon.

"boy_card..." : "girl_card..."` creates two risks: (1) parent profiles get child frames, (2) future non-binary assets (K-01) must be patched in 15+ places instead of 1.

| `ParentDashboardView.swift` | 1 | DashboardChildTab |
| `AvatarCreatorView.swift` | 1 | AvatarPreviewView.characterImage |
| `SiblingProfileView.swift` | 1 | characterImage |

### P1-C: Missing `.trailingFade()` on horizontal carousels (Rule §4 + Anti-pattern C/E)

`.trailingFade()` is the mandatory at-rest scroll cue for horizontal carousels (per §4: "iOS's default scrollbar only appears mid-gesture; kids don't intuit swipe without this"). Missing it means children don't know content exists off-screen.

| File | Missing from |
|---|---|
| `GardenView.swift` | 3 horizontal ScrollViews (seed selection, quick actions, plant type picker) |
| `KitchenView.swift` | 2 horizontal ScrollViews (category tabs, recipe row) |
| `ChefAcademyApp.swift` | 1 ScrollView (sibling card row) |

Note: `FarmShopView.swift` category pills ✓ and `RecipeCardExample.swift` category pills ✓ — already correct.

**Fix:** Add `.trailingFade()` modifier (from `AdaptiveLayout.swift`) to each affected `ScrollView` or its containing `HStack`.

---

### P1-D: "Buy & Plant" with no confirmation dialog (Anti-pattern A/I)

`PlantingSheet.swift` presents a "Buy & Plant" primary CTA that immediately spends coins and plants a seed with no confirmation step. For a 6-year-old who can't read well, accidental taps consume in-game currency irreversibly. `PipDialogView` exists precisely for this pattern ("Spend N coins and plant?").

**Fix:** Gate the "Buy & Plant" action behind `PipDialogView` with coin cost displayed prominently. `BouncyButtonStyle` on the confirm button.

---

### P1-E: KitchenView — No persistent cooking phase indicator (Anti-pattern C)

`KitchenCookingPhase` tracks the user's progress through recipe stages (prep → cook → plate) but there is no persistent ambient indicator in `KitchenView` showing which phase the player is in. When a child returns to the kitchen tab, they cannot tell at a glance what comes next without re-reading text. The "Next Step" affordance must be visible at all times.

**Fix:** Add a persistent step indicator (e.g., 3-dot progress pips using `Color.AppTheme.goldenWheat` for complete, `Color.AppTheme.sage` for current, `Color.AppTheme.parchment` for upcoming) anchored to the top of `KitchenView`.

---

### P1-F: `CookingCompletionView` — Button order reversed (Anti-pattern L)

The primary action ("Cook Again" / "Done") and secondary action are in reversed visual order relative to platform convention and the app's own pattern elsewhere (primary = bottom, most reachable). A child who achieved stars expects a celebratory primary CTA at the bottom reachable zone. Both buttons are also hand-rolled instead of using `.texturedButton(tint:)` and `BouncyButtonStyle`.

**Fix:** Swap button order. Apply `.texturedButton(tint: Color.AppTheme.sage)` to primary, `.buttonStyle(BouncyButtonStyle())` to secondary.

---

### P1-G: `AssembleMiniGame` — Double-fire (Button + `.onTapGesture`) (Anti-pattern G)

`AssembleMiniGame` in `CookingMiniGames.swift` has both a `Button { }` wrapper and a redundant `.onTapGesture { }` on the same ingredient element. SwiftUI processes both, causing `assembleIngredient()` to fire twice per tap — potentially marking an ingredient assembled twice, breaking scoring.

**Fix:** Remove `.onTapGesture`. The `Button` action is sufficient and is the correct interactive container.

---

### P1-H: `CookingSessionView` — No skip mechanic for repeat animations (Anti-pattern C)

Explanation animations in `CookingSessionView` are unskippable on every repetition of a recipe. The UX audit (March 2026) flagged this as a primary annoyance source; the CLAUDE.md roadmap confirms animations should be skippable after first viewing. Currently there is no `hasSeenAnimation(for:)` check or skip affordance.

**Fix:** Track seen animations in `PlayerData` (array of `String` animation IDs, follows the existing `[CodableStruct]` array pattern per Rule §1). On second+ view, show a "Skip ▶" button (`.buttonStyle(BouncyButtonStyle())`). First-view animations play in full.

---

### P1-I: Hand-rolled CTAs — 25+ sites across 10 views

The following views build buttons by composing `.background()` + `.cornerRadius()` + `.shadow()` on a `Text` or `Label` inside a `Button`, bypassing `.texturedButton(tint:)` and `BouncyButtonStyle`. This breaks visual consistency and loses the wood-grain brand identity.

| File | Count |
|---|---|
| `GlucoseJourneyView.swift` | 6 |
| `LocalVersusView.swift` | 3 |
| `SplitScreenVersusView.swift` | 3 |
| `NearbyVersusView.swift` | 4 |
| `MultiplayerHealthyPicksView.swift` | 4 |
| `HealthyChoiceGameView.swift` | 3+ |
| `BodyBuddyView.swift` | 1 ("Cook Something!") |
| `CookingCompletionView.swift` | 2 |
| `PlayLearnView.swift` | 1 ("Back to Games") |
| `SiblingProfileView.swift` | 2 |

**Fix:** Primary CTAs → `.texturedButton(tint: Color.AppTheme.sage)`. Game/bounce CTAs → `.buttonStyle(BouncyButtonStyle())`. Plain card-label buttons → `.buttonStyle(.plain)` only when the label IS the full visual surface.

---

### P1-J: `AllergenPickerStep` — Hand-rolled Back/Next buttons

`AllergenPickerStep.swift` has custom "Back" and "Next" navigation buttons that don't use the project's button components. Because allergen setup is high-stakes (affects all recipe filtering), the CTA design should match the app's established trust pattern.

**Fix:** "Next" → `.texturedButton(tint: Color.AppTheme.sage)`. "Back" → `.buttonStyle(BouncyButtonStyle())` or `.secondaryButton()`.

### SR-4: DispatchQueue → Task pattern (P0 fix — same for all 27 sites)

```swift
// BEFORE (any GKDelegate, MCSessionDelegate, ASAuthDelegate callback)
DispatchQueue.main.async {
    self.isConnected = true

Task { @MainActor in
    self.isConnected = true
    self.route = .mainApp(id)
}
```

For callbacks on classes not already `@MainActor`, add `@MainActor` to the class or the specific method, or use `await MainActor.run { }` inside a non-isolated task.
