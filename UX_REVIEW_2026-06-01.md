## TL;DR

The core GROW → COOK → FEED loop is mechanically complete. The garden, cooking session, Body Buddy, and mini-games all function. What remains are **architecture debt** items that compound each other: `Timer.scheduledTimer` in 7 game-facing files risks broken physics on 120 Hz devices; `PrimaryButtonStyle` (declared dead in STYLES.md) is used in 15 call sites across onboarding and setup screens; 7 horizontal carousels are missing the mandatory `.trailingFade()` scroll cue; and two `§4 profilePoseImage` bypasses create a maintenance fork that will silently break when non-binary avatars land. None of these are visible bugs today, but each one adds cost to future work and creates detectable inconsistency for a 6-year-old user.

## P0 — Blockers (kid-breaking)

### P0-1 · Bug-rescue tap targets critically below 44 pt minimum
**File:** `PlotView.swift` line 392
Bug emoji rendered at `.AppTheme.rounded(size: [12, 10, 11][i])` — the smallest is 10 pt. A child's fingertip covers ~44 pt. The rescue mechanic is currently near-untappable for the age-6+ audience.

**Fix:** wrap each bug emoji in a `Button` with `.frame(minWidth: AppSpacing.minTapTarget, minHeight: AppSpacing.minTapTarget)` and move the bug image inside. Use `.buttonStyle(.plain)` on the wrapper and `.contentShape(Rectangle())` to guarantee the full frame is hittable. Prefer 60 pt for age 6+.

### P0-2 · Timer.scheduledTimer in 7 game-facing files — broken on 120 Hz
**Files:** `PlotView.swift` (line 428), `CookingMiniGames.swift` (lines 121, 746, 932), `ChopMiniGame.swift` (line 53), `HealthyChoiceGameView.swift` (line 609), `FamilySetupView.swift` (lines 610, 634, 668)

`Timer.scheduledTimer` drives physics and frame rate in these files. On ProMotion 120 Hz devices the frame rate doubles; on a CPU-throttled device it halves. The result is visually different game speeds depending on hardware — peel swipes complete instantly on one device and feel sluggish on another. `ANIMATIONS.md` explicitly bans Timer for continuous animation.

**Fix (for each file):** replace Timer-based loops with `TimelineView(.animation) { context in ... }` using delta-time from `context.date`. See the existing correct patterns in `RainOverlay`, `WaterPourCharacterView`, and `InsulinTetrisView`. For `FamilySetupView` avatar animation (lines 610, 634), use `TimelineView(.periodic(from: .now, by: 1.0 / anim.fps))`.

**Exception:** `SessionManager.swift:434` — a 60-second play-time tick is not a visual physics loop and is acceptable as-is.

### P0-3 · "Drag me!" / "Tap me!" hint text at 9 pt — invisible to age 6+
**File:** `GardenView.swift` (DraggablePipView hint label)
The text telling kids to interact with Pip is rendered at ~9 pt (inferred from context — smaller than `.AppTheme.caption` which is 12 pt). A 6-year-old cannot read this. The entire drag-to-harvest mechanic is undiscoverable without adult help.

**Fix:** replace the inline hint label with `PipSpeechBubble` showing "Drag me to a ready plant!" using `.AppTheme.body` (17 pt minimum). Use `speakOnAppear: true` so free-tier users hear it on first garden visit. Gate subsequent shows with `UserDefaults` key `hasSeenDragPipHint` so it appears only on the first 3 visits.

## P1 — Friction

### P1-1 · Seven horizontal carousels missing `.trailingFade()` scroll cue
Kids do not intuit swipe without a visual affordance. iOS's built-in scrollbar only appears mid-gesture — by then the kid has already left the screen thinking there's nothing more.

| File | Location | Has .trailingFade? |
|------|----------|--------------------|
| `GardenView.swift` | Seed inventory carousel (line 1032) | No |
| `GardenView.swift` | Harvested ingredients carousel (line 1119) | No |
| `ChefAcademyApp.swift` | Sibling cards carousel (line 664) | No |
| `HomeAnimated.swift` | Recipe carousel (line 123) | No |
| `KitchenView.swift` | Pantry/recipe scrolls (lines 780, 790) | No |
| `GlucoseJourneyView.swift` | Snack options carousel (line 449) | No |

**Fix:** add `.trailingFade()` immediately after each `.scrollView { ... }` close brace. No other changes needed — the modifier is already defined in `AdaptiveLayout.swift` and handles all color/width logic internally.

### P1-2 · PrimaryButtonStyle used in 15 call sites — declared dead code
**STYLES.md:** "Primary | `.primaryButton()` | Legacy (dead code — used only in own preview); prefer Textured."
**Files with live call sites:** `FamilySetupView.swift` (×7), `MeetPipViews.swift` (×3), `AvatarCreatorView.swift`, `AddChildFlowView.swift`, `OnboardingView.swift` (×2), `MeetPipAnimated.swift` (×2)

These are the **first screens** a new family sees. If `PrimaryButtonStyle` is deleted in a future cleanup pass, 15 screens in the critical first-run funnel will break simultaneously.

**Fix:** replace each `.buttonStyle(PrimaryButtonStyle())` with `.texturedButton(tint: Color.AppTheme.sage)`. This is a mechanical substitution with no layout impact — `TexturedButtonStyle` uses the same capsule shape and button height token.

### P1-4 · Two `§4 profilePoseImage` bypasses create a maintenance fork
| File | Line | Hardcoded expression |
|------|------|----------------------|
| `ChefAcademyApp.swift` | 669 | `sibling.gender == .boy ? "boy_card_clean_frame_11" : "girl_card_clean_frame_06"` |

### P1-5 · Kitchen hidden-mode taps (anti-pattern A) — same button, different actions
**File:** `KitchenView.swift`
Counter, Stove, and Pantry hotspots change behavior completely based on `cookingPhase`. In `browsing` phase they inspect; in `gatherPantry` they add ingredients; in `stoveReady` they start cooking. The visual difference is a subtle label text change. A 6-year-old who taps Counter while in `gatherPantry` phase expecting to inspect gets a surprising add-to-pan action instead.

**Fix:** during `gatherPantry` phase, add a visible badge ring around the active hotspot using `Color.AppTheme.brightGreen.opacity(0.3)` as an animated glow (see `AnimationConstants.floatLoop`). Use `PipSpeechBubble` positioned above the active spot saying "Tap here to gather!" so the kid knows which button to press. The phase change should be accompanied by a `Haptic.impact(.medium)` to signal the mode shift.

### P1-6 · PlayLearnView dead-end screens — tapping opens "Coming Soon!" placeholder
**File:** `PlayLearnView.swift`
Three unlocked-looking game cards ("Veggie Match", "Nutrition Quiz", "Chop Challenge") open placeholder views with only a "Back to Games" button. Kids who tap expecting a game experience a hard dead-end with no explanation. This is anti-pattern H (missing next-step cue) combined with anti-pattern J (scary/confusing feedback).

**Fix:** mark these cards with the same `.disabled(true)` and "Coming Soon!" overlay as the other locked games, OR replace the placeholder view with `PipSpeechBubble("This game is being built! Check back soon 🌱")` plus an animated Pip pose. Do not present an empty screen.

### P2-5 · BodyBuddyView health orb positions use hardcoded pixel offsets
**File:** `BodyBuddyView.swift`
Organ positions use `.offset(y: -110)`, `.offset(x: -70, y: -40)` etc. — raw pixel values not tied to any token. These will misalign on any device size class not tested during development. Prefer `AdaptiveCardSize` constants or fraction-based positioning relative to the body image frame.

### P2-6 · DispatchQueue.main.async in framework bridge files
The §2 rule bans `DispatchQueue.main.asyncAfter` but also requires `Task { @MainActor in }` for delayed mutations. Non-delayed `DispatchQueue.main.async { }` callbacks in UIKit bridge files technically violate the spirit of the rule:
- `AuthManager.swift:113`
- `GameCenterMatchmakerView.swift` (×3)
- `GameCenterService.swift:102`
- `MultiplayerManager.swift` (×5)

These are in UIKit delegate callbacks (`GKMatchmakerViewControllerDelegate`, `ASAuthorizationControllerDelegate`) where the framework calls back on an unspecified thread. `DispatchQueue.main.async` is the correct UIKit interop pattern here. However, the newer `MainActor.assumeIsolated { }` or `await MainActor.run { }` pattern is preferred under strict Swift 6 concurrency. Low risk in current form; worth noting for a future Swift 6 migration.

### P2-7 · `try? modelContext.save()` in 8 locations — silent failure risk
**Files:** `AddChildFlowView.swift:132`, `ChefAcademyApp.swift:464`, `FamilySetupView.swift:204`, `AllergenEditorSheet.swift:119`, `SessionManager.swift` (lines 100, 206, 355, 419), `ParentDashboardView.swift:461`

§1 Architecture Rule explicitly states: "Use `do { try save() } catch { print(error) }` for SwiftData saves — never `try?`. Silent failures destroyed child profiles for a week (March bug)."

**Fix:** wrap each `try? modelContext.save()` in `do { try modelContext.save() } catch { print("SwiftData save failed: \(error)") }`.
