## Section 2 — TL;DR

**3 P0s, 12 P1s, 17 P2s.**

The biggest single issue — one that touches 13 active-code call sites — is `profilePoseImage` bypassed in favour of raw gender conditionals. Parents see kid frames instead of the mom/dad frames designed for them, silently breaking the parent visual identity across every multiplayer screen. The second most systemic issue is `PrimaryButtonStyle()` (dead code) still used in 8+ onboarding files instead of `.texturedButton(tint:)`, guaranteeing the wrong visual weight on every first-launch CTA a new user ever sees. Third is PlotView's 10–12pt bug-emoji tap targets — a 6-year-old literally cannot hit them, making the most time-sensitive garden interaction a frustrating miss-fest.

Bright spots: the multiplayer physics engine (TimelineView + delta-time throughout), `PipDialogView`'s gift confirmation guard, the sticky "Let's Cook!" footer in `RecipeDetailView`, and the whole `SubscriptionManager` / `AppAttestService` stack — all clean, modern, production-quality code.

## Section 3 — P0 Findings (Blockers)

### P0-01 · D-01 · PlotView — Bug tap targets below minimum
**File:** `PlotView.swift`
**Symptom:** The "Tap bugs!" mini-game uses emoji characters rendered at font size 10–12pt as the tap targets. The care hint labels ("Hold me!", "Swipe up!", "Tap bugs!") are also 10pt. Both are well below the `AppSpacing.minTapTarget` of 44pt. A 6-year-old will miss every bug tap and receive no feedback, turning the most time-sensitive garden interaction into an invisible failure loop.
**Fix:** Wrap the bug emoji in a `Button` with an explicit `.frame(width: AppSpacing.minTapTarget, height: AppSpacing.minTapTarget)` tap area. Raise hint labels to `Font.AppTheme.caption` (12pt minimum) or `Font.AppTheme.subheadline` (15pt preferred). Use existing tokens only — no inline sizes.

### P0-02 · C-01 · GardenView — Contradictory hint instruction
**File:** `GardenView.swift` (`gardenHint`)
**Symptom:** The garden hint reads "Drag Pip to a glowing plot to harvest!" — but Pip was removed from the garden map. The instruction describes a mechanic that no longer exists. A new 6-year-old player will look for Pip to drag, find nothing draggable, and assume the app is broken.
**Fix:** Update `gardenHint` to describe the actual tap-to-open-plot affordance (e.g., "Tap a glowing plot to care for your plant!"). Use existing font/color tokens. Zero source-file edits to CLAUDE.md or AppTheme needed.

### P0-03 · L-01 · PrimaryButtonStyle() dead code on first-launch CTAs
**Files:** `AddChildFlowView.swift`, `AvatarCreatorView.swift`, `MeetPipViews.swift`, `FamilySetupView.swift` (FamilyWelcomeStep), `MeetPipAnimated.swift` (x2), `OnboardingView.swift` (x2) — 8+ occurrences
**Symptom:** The very first CTAs a new user ever sees — "Let's Go!", "Start!", "Next" in onboarding and family setup — use `PrimaryButtonStyle()`, which is confirmed dead code (no texture, wrong visual weight). Per Architecture Rule §4, primary CTAs must use `.texturedButton(tint:)` (wood-grain capsule). The mismatch means the signature botanical look is absent at the highest-stakes moment in the funnel.
**Fix:** Replace every `PrimaryButtonStyle()` occurrence with `.texturedButton(tint: Color.AppTheme.sage)`. Where the tint should differ (e.g., golden CTA), use `.texturedButton(tint: Color.AppTheme.goldenWheat)`. No new tokens needed.

## Section 4 — P1 Findings (Significant UX Degradation)

### P1-01 · C-02/C-03/C-04 · Missing .trailingFade() on three carousels
**Files:** `GardenView.swift` (seed inventory carousel, harvested ingredients carousel), `ChefAcademyApp.swift` (~line 669, siblings carousel in HomeView)
**Symptom:** All three horizontal carousels have `showsIndicators: false` with no scroll cue. Per Architecture Rule §4, horizontal carousels must use `.trailingFade()` as the at-rest scroll affordance. iOS's default scrollbar only appears mid-gesture. A 6-year-old won't intuit swipe without it. ParentDashboardView already applies this correctly — HomeView siblings and GardenView carousels do not.
**Fix:** Add `.trailingFade()` to the three missing carousels. Token exists in `AdaptiveLayout.swift`.

### P1-02 · C-05 · KitchenView — "Farm Shop" text is non-tappable
**File:** `KitchenView.swift` (almostReadySection)
**Symptom:** When a recipe is almost-ready, the view shows "Need more from the Farm Shop" as plain `Text`. The child reading this cannot tap it to navigate. There is no button, no navigation link, no tap gesture. A kid who wants to follow the instruction is stuck.
**Fix:** Wrap "Farm Shop" in a `Button` that routes to `Tab.shop` via `sessionManager.selectedTab` (or equivalent route). Style with `foregroundColor(Color.AppTheme.goldenWheat)` + `.underline()` for tappability affordance. No new tokens needed.

### P1-03 · L-02 · BodyBuddyView — "Cook More!" violates Architecture Rule §4
**File:** `BodyBuddyView.swift`
**Symptom:** The "Cook More!" CTA uses `.buttonStyle(.plain)` with a manual `.background(Color.AppTheme.sage)` + `.cornerRadius(AppSpacing.cardCornerRadius)`. This is the exact pattern explicitly banned by Architecture Rule §4: "Never `.buttonStyle(.plain)` with a custom-styled label." The result is visual weight inconsistency — it looks like a button but has no bounce feedback or texture.
**Fix:** Replace with `.buttonStyle(BouncyButtonStyle())` and let the label styling drive the appearance, or use `.texturedButton(tint: Color.AppTheme.sage)` if it is a primary CTA.

### P1-04 · D-02 · profilePoseImage bypassed in 13 active-code locations
**Files:**
- `ChefAcademyApp.swift` ~line 669 (siblings section of HomeView)
- `AvatarCreatorView.swift` line 114
- `SiblingProfileView.swift` line 26
- `LocalVersusView.swift` lines 199, 404, 437, 463
- `MultiplayerHealthyPicksView.swift` lines 358 (opponentScoreBar), 600 (playerAvatar)
- `NearbyVersusView.swift` lines 275 (opponent HUD), 525 (playerAvatar helper)
- `SplitScreenVersusView.swift` lines 111 (pickPlayers grid), 194 (readyView P1), 215 (readyView P2)

**Symptom:** All 13 sites use the hardcoded pattern `gender == .boy ? "boy_card_clean_frame_11" : "girl_card_clean_frame_06"` instead of the `UserProfile.profilePoseImage` helper. The helper routes parents to `mom_avatar_frame_15` / `dad_avatar_frame_15`. Without it, a parent profile always renders with child frames — the visual identity system built to distinguish parents from children silently breaks.
**Fix:** Replace each inline conditional with `profile.profilePoseImage` (or `sibling.profilePoseImage`, `player.profilePoseImage` as appropriate). One-token-per-site change, zero new code.

### P1-05 · H-01 · FamilyMeetPipStep — 3-dialog sequence with no progress indicator
**File:** `FamilySetupView.swift` (FamilyMeetPipStep)
**Symptom:** Three Pip speech bubbles appear sequentially with no "1 of 3" indicator. A child (or parent) has no way to know how long this section lasts. The March 2026 audit identified indefinite-length sequences as an engagement cliff — users tap away before reaching the CTA because they can't assess commitment.
**Fix:** Add a `ProgressView(value: Double(dialogIndex + 1), total: 3.0)` above the bubble. Style with `.tint(Color.AppTheme.sage)`. No new tokens needed.

### P1-06 · G-01 · FarmShopView — Double confirmation UX
**File:** `FarmShopView.swift`
**Symptom:** After the PipDialog "Spend N coins?" confirm fires, a second `.alert("Purchased!")` also fires. The child taps confirm, is shown a Pip dialog, then is immediately shown a system alert. Two successive confirmations is twice the friction and creates confusion about what just happened.
**Fix:** Remove the secondary `.alert("Purchased!")`. The PipDialog confirmation is sufficient. The coin balance change is immediate visual feedback.

### P1-07 · F-02 · KitchenView hint at caption size
**File:** `KitchenView.swift` (`kitchenHint`)
**Symptom:** The Kitchen tab's contextual hint — which describes the most complex multi-step flow in the app — renders at `.caption` (12pt). For a 6-year-old navigating cooking for the first time, this is the least accessible hint at the highest-complexity moment.
**Fix:** Raise to `Font.AppTheme.subheadline` (15pt). Token exists.

### P1-08 · B-01 · KitchenView — 4+ steps before cooking begins
**File:** `KitchenView.swift`
**Symptom:** To cook a recipe, a child must: (1) tap Kitchen tab, (2) find a recipe in the list, (3) open RecipeDetailView, (4) review ingredients, (5) tap "Let's Cook!" — five sequential steps before the first mini-game appears. The UX Redesign Plan mandates a 4-step ceiling for any process. This flow is one step over.
**Fix:** Collapse step 4 (ingredient review) into step 3 by surfacing the ingredient check inside RecipeDetailView inline (already partially done via the ingredient list), or add a "Quick Cook" shortcut from KitchenView's ready-recipe card directly to CookingSessionView. No new tokens needed.

### P1-09 · Architecture · Inline animation violations (16 files)
**Files:** `GameState.swift`, `FamilySetupView.swift`, `GlucoseJourneyView.swift`, `AskPipView.swift`, `BodyBuddyView.swift`, `ChopMiniGame.swift`, `CookingMiniGames.swift` (8 instances), `CookingSessionView.swift` (line 523), `HealthyChoiceGameView.swift` (lines 411, 812), `PipAnimations.swift` (WiggleModifier line 489), `MeetPipAnimated.swift` (ConfettiView line 381), `PipTestView.swift` (line 162)
**Symptom:** Inline `.easeInOut(duration:)`, `.easeOut(duration:)`, `.easeIn(duration:)`, and `.spring(response:)` calls throughout active source. Per Architecture Rule §3, all animation curves must go through `AnimationConstants.*`. These inline values will drift from the design system over time and break animation coherence on ProMotion displays.
**Fix per file:** Replace each inline curve with the nearest `AnimationConstants` match:
- `easeOut(duration: 0.3)` → `AnimationConstants.springQuick`
- `easeInOut(duration: 0.4)` → `AnimationConstants.fadeMedium`
- `.spring(response: X)` → `AnimationConstants.springMedium` or `.springBouncy`
- `easeIn(duration: 1.5+)` (long fades) → `AnimationConstants.revealSlow`
- `WiggleModifier` needs a new `AnimationConstants.wiggle` constant (add to `AppTheme.swift`).

### P1-10 · Architecture · Color.black.opacity() shadow violations
**Files:** `GardenView.swift` (DraggablePipView, WalkingPipView), `ChopMiniGame.swift`
**Symptom:** `.shadow(color: Color.black.opacity(N))` used directly. Per STYLES.md, shadows must use `Color.AppTheme.sepia.opacity(N)` — never `Color.black`. The result is harsher, greyer shadows that break the botanical watercolor aesthetic.
**Fix:** Replace `Color.black.opacity(N)` with `Color.AppTheme.sepia.opacity(N)` in each shadow. No new tokens needed.

### P1-11 · Architecture · .font(.system(size:)) inline
**File:** `ProfilePickerView.swift` lines 39, 86
**Symptom:** Two inline font definitions bypass `Font.AppTheme`. Per Architecture Rule §3, inline `.font(.system(size:))` is banned. These are on the "Who's playing today?" screen — the first screen every child sees after family setup.
**Fix:** Replace with the nearest `Font.AppTheme` token (`caption` at 12pt, `subheadline` at 15pt, or `Font.AppTheme.rounded(size:weight:)` if a custom rounded size is needed).

### P1-12 · Architecture · DispatchQueue.main.async violations
**Files:** `ParentPINEntryView.swift` (startAppleIDVerification), `AuthManager.swift` (line 113, checkExistingCredential), `GameCenterService.swift` (line 102, authenticate), `MultiplayerManager.swift` (authenticateLocalPlayer line 65, handleMessage line 196, startCountdown lines 241-251, GKMatchDelegate lines 297/321), `NearbyMultiplayerManager.swift` (handleMessage line 155, startCountdown lines 198-209, MCSessionDelegate line 241, advertiser line 287, browser lines 307/311)
**Symptom:** `DispatchQueue.main.async { }` used in Timer callbacks and delegate callbacks instead of `Task { @MainActor in }`. Per Architecture Rule §2, all UI state mutations from callbacks go through `Task { @MainActor in }`. The delegate callback pattern is particularly risky under Swift 6 strict concurrency — these will produce data-race warnings.
**Fix:** Replace each `DispatchQueue.main.async { ... }` block with `Task { @MainActor in ... }`. The inner content is unchanged.

## Section 7 — Suggested Redesigns (Top 3)

### Redesign 1 · PlotView — Accessible bug-tap targets (P0-01)

**Problem:** Bug emoji at 10–12pt; hint labels at 10pt. Misses create a dead-feedback loop for 6-year-olds.

**Proposed Change:**
```swift
// BEFORE (PlotView — approximate current pattern):
Text("🐛")
    .font(.system(size: 12))           // ← banned, <44pt, invisible tap zone
    .onTapGesture { rescueBug() }

Text("Tap bugs!")
    .font(.system(size: 10))           // ← banned, illegible

// AFTER:
Button(action: rescueBug) {
    Text("🐛")
        .font(.AppTheme.title)         // 28pt emoji — visible
}
.buttonStyle(BouncyButtonStyle())
.frame(
    width: AppSpacing.minTapTarget,    // 44pt — minimum tap area
    height: AppSpacing.minTapTarget
)

Text("Tap bugs!")
    .font(.AppTheme.subheadline)       // 15pt — legible for 6+
    .foregroundColor(Color.AppTheme.darkBrown)
```
**Tokens used:** `AppSpacing.minTapTarget`, `Font.AppTheme.title`, `Font.AppTheme.subheadline`, `Color.AppTheme.darkBrown`, `BouncyButtonStyle()` — all existing.

### Redesign 2 · GardenView — Fix contradictory hint + add scroll cues (P0-02 + P1-01)

**Problem:** gardenHint describes a removed mechanic; seed + harvested carousels have no scroll cue.

**Proposed Change:**
```swift
// BEFORE (GardenView):
let gardenHint = "Drag Pip to a glowing plot to harvest!"   // ← Pip removed!

// Seed carousel — no scroll cue
ScrollView(.horizontal, showsIndicators: false) {
    HStack { ForEach(seeds) { ... } }
}

// AFTER:
let gardenHint = "Tap a glowing plot to water, weed, and harvest!"

// Seed carousel — with scroll cue
ScrollView(.horizontal, showsIndicators: false) {
    HStack { ForEach(seeds) { ... } }
}
.trailingFade()     // ← existing AdaptiveLayout modifier

// Harvested carousel — same pattern
ScrollView(.horizontal, showsIndicators: false) {
```
